## 进度更新（2026-06-19）：昨天的卡点已解决
昨天怀疑的「loopback 延迟注入 bug / one-shot timer」**是误判**——延迟注入机制本身正常。主 agent 实跑后定位到**两个互相独立的真 bug**:

1. **测试 harness 孤儿串台(主 agent 修)**:`ros2 run` 把节点拉成子进程,`kill` 只杀父包装进程,真节点变孤儿继续在 DDS 图广播;`pkill -f <节点名>` 还会匹配脚本自己导致清理自杀(exit 144)。多个孤儿 exo_cmd 各自从 0 发心跳互相串台 → 指标爆炸(8s 内 sent=6000+、duplicate 8 万),长得像逻辑 bug 实为环境污染。**昨天的 matched=0 主要是这个。** 已在 `tools/run-scenario.sh` 用 **setsid + 按进程组 `kill -- -PGID` + 跑前 `pgrep` 硬断言无孤儿**根治,复跑实测零孤儿。教训已存记忆 `ros2-verify-kill-process-group-or-crosstalk`。
2. **`loopback_node.py` 的 `drop_seqs` 声明崩溃(Tom 修)**:空列表默认值被 rclpy Jazzy 锁成 BYTE_ARRAY,传整型数组就崩、节点起不来 → A3 确定性丢包从声明之日起没真正可用过(逻辑单测照不出)。Tom 改用 `ParameterDescriptor(dynamic_typing=True)`,已 rebuild。

## 重要环境约束(踩过的坑)
- **sudo 装包必须用户在交互终端手动跑**:agent 会话无 TTY 输不了 sudo 密码,所有 `apt` 安装都由用户执行脚本完成。
- **子 agent 的 Bash 被沙箱大幅收紧**:实测连含 `source` 的命令都被拦(即便配置已 allow);但 **主 agent 的 Bash 可放行 `colcon`/`ros2`/`source`** —— 所以"build/launch/实跑验证"这类执行交给主 agent 做,子 agent(Tom)专注写代码落盘。这是本项目稳定的分工模式。
- env-check.sh / 涉及 ROS 的脚本 source 前需 `set +u`(ROS setup.bash 引用未初始化变量,与 set -u 冲突)。
- `ros2 topic info -v` 的 History(Depth) 恒显 UNKNOWN(rmw 发现不传 depth),非 QoS 缺陷;以代码里的 QoSProfile 为准。
- 同一 WSL 上跑多份 launch 会在默认 DDS 图里串台(topic 同名互联),验证时确保只有一份实例。
