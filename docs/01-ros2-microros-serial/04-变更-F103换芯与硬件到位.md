# 变更设计：F401 → F103 换芯 + WSL 硬件到位（Phase B 解锁）

> 作者：elon（设计），主 agent 代笔落盘（2026-06-19）。本文是针对用户两点决策（MCU 改 STM32F103 / Nucleo-F103RB；板子已通过 USB 接到运行 WSL 的 Windows）的增量评估与任务卡增量。其余已锁决策（ROS2 Jazzy、裸 CMake + arm-none-eabi-gcc、FreeRTOS、micro-ROS over serial、921600、DMA、双向 pub/sub）全部不变。

## 0. 一句话结论
换芯本身在工具链、引脚、时钟、波特率四个维度都是**低风险、可执行的小改动**（A.1/A.3/A.4）。**唯一的真风险、也是项目级风险，是 RAM：F103RB 只有 20KB SRAM，而 micro-ROS 官方推荐最低 32KB。** 这不是“调参就稳”的事，有翻车可能。下文给出“尽力压”方案 + 明确降级预案，并建议把“能否在 20KB 内跑起来”作为一个**独立的、提前做的去风险 spike（里程碑 T8）**，而不是埋在固件大卡里到最后才爆。

## A. F401 → F103 关键差异及影响

### A.1 内核 Cortex-M4F → Cortex-M3（无 FPU）
- **工具链 flag 必须改**（Tom 工具链/骨架卡的硬改动）：

  | flag | F401（旧 M4F） | F103（新 M3） |
  |------|---------------|---------------|
  | `-mcpu` | `cortex-m4` | **`cortex-m3`** |
  | `-mfpu` | `fpv4-sp-d16` | **不设（M3 无 FPU）** |
  | `-mfloat-abi` | `hard` | **`soft`** |
  | `-mthumb` | 有 | 有（不变） |

  坑：残留 `-mfloat-abi=hard` 或 `-mfpu=...` 会导致 ABI 不匹配链接失败/行为错误；三个 flag 必须一起改。
- **float 现在是软浮点**，M3 上走 libgcc 仿真，慢一两个数量级。当前 Int32 回环不涉及浮点，无影响；**固件路径上禁用浮点**（RTT 监控在 WSL 侧算，固件只搬字节），同时利好 RAM。
- 启动文件/链接脚本换 `STM32F103xB`（Flash 128K / RAM 20K）；HAL 从 `stm32f4xx_hal` 换成 `stm32f1xx_hal`。**F1 与 F4 的 GPIO/USART/DMA HAL API 有差异**（F1 是输入/输出模式 + CNF 老模型 + AFIO 重映射；F4 是 MODER/AFR per-pin AF）。Tom 须按 F1 HAL 重写初始化，不能照抄 F4 的 `MX_USART2_UART_Init`。

### A.2 RAM 20KB / Flash 128KB —— 最大风险点（重点）
事实：F103RB **SRAM 20KB / Flash 128KB**（F401RE 是 ~96KB/512KB，RAM 缩到约 1/5，是质变）。micro-ROS 官方推荐**至少 32KB RAM**，20KB 低于下限；社区（micro_ros_stm32cubemx_utils issue #35）正是在 F103RB 上踩到 SRAM 溢出，且**无成功收尾记录**。

20KB 要同时容纳：① micro-ROS client + XRCE-DDS 中间件内存，② FreeRTOS 内核 + 任务栈（micro-ROS 任务栈是大头），③ XRCE-DDS serial 收发 buffer，④ USART DMA 双缓冲，⑤ HAL/系统 .data/.bss。默认配置几乎必然溢出，**能否跑下取决于把中间件裁到多窄**。

**“尽力压”方案（F103 专用 colcon.meta 关键裁剪，键已据 micro_ros_mbed 官方 colcon.meta 核实可行）：**
```
rmw_microxrcedds:
  cmake-args:
    - -DRMW_UXRCE_MAX_NODES=1
    - -DRMW_UXRCE_MAX_PUBLISHERS=1          # 只有 mcu_status
    - -DRMW_UXRCE_MAX_SUBSCRIPTIONS=1       # 只有 cmd_heartbeat
    - -DRMW_UXRCE_MAX_SERVICES=0
    - -DRMW_UXRCE_MAX_CLIENTS=0
    - -DRMW_UXRCE_MAX_HISTORY=1             # 历史深度压到 1（与契约 Depth=10 有张力，见下）
    - -DRMW_UXRCE_XML_BUFFER_LENGTH=400
    - -DRMW_UXRCE_TRANSPORT=custom          # 自己的 USART+DMA custom transport
    - -DUCLIENT_PIC=OFF
# client/deps：关闭 filesystem / thread / 64位原子等不需要能力
```
> **实现修正（T-meta，2026-06-19，已落盘）**：上面这份键列表是设计草案；**经 Tom 对 4 个官方源逐键核实后的可构建版在 `firmware/f103-microros/colcon.meta`（+ `.annotated` 注释版 + `README.md`）**，相对草案有两处修正：① `RMW_UXRCE_XML_BUFFER_LENGTH=400` 在当前 rmw_microxrcedds(jazzy) CMakeLists 选项表里**查无此键**，改用 `RMW_UXRCE_CREATION_MODE=refs` 从根上省 XML 缓冲（refs 在 serial 点对点的可行性待 T8 实测，跑不通再回退 bin 并核实 XML_BUFFER 键名）；② 草案漏了两个对 20KB 是大头的键已补：`RMW_UXRCE_STREAM_HISTORY=1`（默认 4）、`UCLIENT_CUSTOM_TRANSPORT_MTU=128`（默认 512）。串口内存预算实测 <1KB（约 640B），详见该目录 README。

配套：micro-ROS 用 **static memory pool / 关动态分配**；任务栈调到“刚好够”（实测 `uxTaskGetStackHighWaterMark` 逐步收，别用默认 10KB）；XRCE-DDS serial buffer 与 DMA 双缓冲共用尽量小的 MTU（容下 Int32+XRCE 头数十字节 + 裕度，DMA RX 双缓冲每个 2×128B 或更小），**整个串口收发占用目标 < 1KB**。

**与现有契约的张力（须显式记录）**：契约 v1.1 要求两条 topic QoS = RELIABLE / KEEP_LAST / **Depth=10**。20KB 上 Depth=10 历史缓存 + RELIABLE 重传缓存是 RAM 杀手，`RMW_UXRCE_MAX_HISTORY=1` 与之冲突。
- **处置建议**：固件侧（micro-ROS client）历史深度降到 1；WSL（ROS2）侧维持 Depth=10；RELIABLE 两端保留。代价是 MCU 侧缓存浅、抖动时更易丢——但这**应被契约第 7 节监控如实暴露**，符合“零容忍=不掩盖”，不违背安全哲学。**这是契约级取舍，须 elon/主 agent 在契约写明理由，不由 Tom 决定。**

**初判**：谨慎乐观但不保证。压到 pub/sub=1、history=1、static pool、小 MTU、砍浮点后社区有 20KB 级极简 pub/sub 跑通先例，但官方不背书、issue #35 无成功记录，**存在压不进的真实可能**，故必须有预案。

**降级预案（按优先级）**：
1. 首选：按上述裁剪压进 20KB，以 `.map` 实测 RAM/Flash + 运行时栈水位为准（“编译过”不算赢，要留运行裕度）。
2. 降级 A — 砍功能：双向压不进就先做**单向**验证 micro-ROS 在 F103 起得来，再逐步加。
3. 降级 B — 换传输/换 client 形态（研究项，不轻动）。
4. 降级 C（架构级，**需用户拍板**）：20KB 实在跑不动 micro-ROS，则回退到“自定义轻量串口协议 + WSL 侧桥接节点”，放弃在 F103 跑 micro-ROS client。这偏离原始需求，**属需求级变更，回来找用户，不由 Tom 私自决定**。

> 建议把 A.2 做成**前置去风险 spike（T8）**，在完整固件开发前先回答“20KB 跑不跑得下”，避免最后才发现要架构回退。

### A.3 时钟树 / 921600 波特率
- F103 最高 **72MHz**；Nucleo-F103RB 板载 **8MHz HSE**（来自 ST-Link MCO），PLL×9=72MHz。**USART2 挂 APB1（最高 36MHz=72/2），USART1 挂 APB2（72MHz）。** 波特率分频用各自总线时钟。F1 只有 16× 过采样（921600 不需 8×）。
- 手算误差（16×，USARTDIV=f_PCLK/(16×baud)）：

  | 外设 | 总线 | USARTDIV | BRR(M/F) | 实际波特率 | 误差 |
  |------|------|----------|----------|-----------|------|
  | **USART2（用）** | APB1 36MHz | 2.4414 | 2 / 7 → 2.4375 | 923077 | **+0.16%** |
  | USART1（备选） | APB2 72MHz | 4.8828 | 4 / 14 → 4.875 | 923077 | **+0.16%** |
- 结论：**误差约 +0.16%，远低于 ~2.5% 容限，无需改波特率。** 备选（将来需要）：460800 / 115200。
- 提醒 Tom：PLL 锁 72MHz、APB1 分频=2、APB2 分频=1、Flash 2 等待周期。BRR 由 HAL 按 `HAL_RCC_GetPCLK1Freq()` 自动算，**前提是时钟树配对**，否则按错误 PCLK 算出错 BRR——“波特率对不上”最常见根因。

### A.4 USART2 引脚 / ST-Link VCP（据 UM1724 核实）
- **默认 SB13 ON / SB14 ON、SB62 OFF / SB63 OFF**：USART2 经 ST-Link 提供虚拟串口 VCP。
- USART2 = **PA2(TX) / PA3(RX)**（与 F401RE 相同，Nucleo-64 布局一致）。
- **插一根 USB 线即得 VCP**，MCU USART2 经板载 ST-Link 透传，主机侧为一个串口（WSL 里 `/dev/ttyACM0`），**无需额外 USB-TTL**，与原 F401 方案一致。用户现在这根已插好的线默认就是这条链路。
- 改外接：SB13/14 OFF、SB62/63 ON，PA2(D1)/PA3(D0) 引到 Arduino 排针；**备选独立串口 USART3 = PC10(TX)/PC11(RX)**（替换原 F401 方案里的 USART6）。

**结论**：架构第 4 节 / 契约第 3 节的 “USART2=PA2/PA3、经板载 ST-Link VCP、一根 USB 即通” 对 F103RB 仍成立，引脚不变。只需 board 名 F401RE→F103RB、备选 USART6→USART3。

## B. WSL 串口接入（usbipd-win，从“已插 Windows”到 WSL 见 /dev/ttyACM0）

WSL2 不自动继承 Windows 的 USB 串口，需 usbipd-win 透传。

### B.1 一次性准备
- **[Windows 管理员 PowerShell]** `winget install --interactive --exact dorssel.usbipd-win`（或 GitHub release msi），装完重开 PowerShell。
- **[WSL]** 确认内核带 `cdc_acm`（Ubuntu 24.04 WSL2 开箱可用，用户当前能跑 ROS2，大概率 OK）。

### B.2 每次接板/重启后的流程
1. **[Win 管理员 PS]** `usbipd list` → 找 “STMicroelectronics STLink / STLink Virtual COM Port”，记 **BUSID**（如 `2-3`）与 VID:PID（ST-Link 常 `0483:374b`）。
2. **[Win 管理员 PS，一次性]** `usbipd bind --busid <BUSID>`（持久；以 list 显示 “Shared” 为准）。
3. **[Win PS，每次用]** `usbipd attach --wsl --busid <BUSID>`（指定发行版 `--wsl Ubuntu-24.04`；拔插/重启需重做；新版支持 `--auto-attach`）。
4. **[WSL]** `ls -l /dev/ttyACM*` → ST-Link VCP 一般枚举为 **`/dev/ttyACM0`**（板载 VCP 是 ACM，不是 ttyUSB）。`dmesg | tail` 看 `cdc_acm ... ttyACM0`。
5. **[WSL 权限]** 临时 `sudo chmod 666 /dev/ttyACM0`；正规 `sudo usermod -aG dialout $USER`（重开 shell 生效）。
6. **[WSL]** `ros2 run micro_ros_agent micro_ros_agent serial --dev /dev/ttyACM0 -b 921600 -v6`。

### B.3 收尾 / 排障
- 用完 `usbipd detach --busid <BUSID>`。
- 设备号以 `ls /dev/ttyACM*` 实际为准，agent `--dev` 跟着改。
- **常见坑**：① attach 必须**管理员** PowerShell；② 同一 ST-Link 不能被 Windows 程序（CubeProgrammer/串口助手）与 WSL 同时占用——**烧录（SWD）与 VCP 串口共用同一 USB**。
- **关键提醒**：ST-Link 同时是**烧录器(SWD)** 和 **VCP 串口**，走同一 USB。attach 进 WSL 后**烧录也得在 WSL 内做（st-flash/openocd，T1 已装）**；想用 Windows CubeProgrammer 烧得先 detach。建议统一在 WSL 内烧录 + 联调，避免反复 attach/detach 抢设备。写进真机联调卡。

## C. 任务卡增量 + Gill 验收 + Phase B 提前度 + 下一步顺序

### C.1 修改的现有 Tom 卡
| 卡 | 改动 | 要点 |
|----|------|------|
| **T1（工具链，已完成）** | 小幅回补 | gcc 不分芯片无需重装；toolchain file 目标 M4F→M3（见 T4）。**新增：验证 usbipd-win 链路可用**（B 节）。 |
| **T4（固件骨架）** | 重写为 F103 | flag→cortex-m3/无 fpu/soft（A.1）；startup/linker 换 F103xB；HAL→F1 并按 F1 模型重写 GPIO/USART/DMA；时钟 HSE8M→PLL×9=72MHz/APB1=36M/APB2=72M/Flash 2WS；验收升级为 **.map 实测 RAM/Flash 留裕度**。 |
| **T5（micro-ROS 应用）** | 加 RAM 裁剪 + 砍浮点 | 落实 custom USART+DMA transport，MTU/DMA 双缓冲压小（A.2），固件禁 float。 |
| **T7（真机联调，Phase B）** | 大幅提前 + 补 usbipd/烧录流程 | 硬件到位，T7 不再是遥远最后一步；补 B 节流程与 ST-Link 烧录/VCP 争用协调。 |
| **架构第4节 / 契约第3节** | board 名 + 备选串口 | F401RE→F103RB；备选 USART6→USART3(PC10/PC11)；PA2/PA3 与 VCP 不变。 |
| **契约 QoS（第1节）** | 契约级取舍待定 | F103 侧 history 深度可能降到 1（A.2），与 Depth=10 冲突，由 elon/主 agent 起草修订标注理由。 |

### C.2 新增 Tom 卡
- **T8（建议最先做）—— “20KB 可行性去风险 spike”**：用最裁剪 colcon.meta（A.2）+ 最小 FreeRTOS + 最小 micro-ROS（哪怕只 1 个 pub）在 F103 上编译+烧+真机起 session，以 .map + 运行时栈水位回答“20KB 跑不跑得下”。**产出是 go/no-go 判断，不是完整功能。** 触发降级预案的判定在此。
- **T-meta —— micro-ROS colcon.meta F103 裁剪**：产出 F103 专用 colcon.meta + static memory pool 配置 + 任务栈调优记录，供 T5/T8 共用。

### C.3 Gill 验收点增量
- **G-T4-a**：`arm-none-eabi-readelf -A` 确认产物为 M3，无 FPU/VFP 属性（无残留 hard-float）。
- **G-T4-b**：从 `.map`/`size` 读出 **RAM < 20KB 且有明确裕度**（给实际 KB 数）；Flash < 128KB。
- **G-T8-a（去风险卡验收）**：真机 micro-ROS session 能建立（agent `-v6` 见 session created）；跑 N 分钟无栈溢出/hardfault；给 RAM 水位实测。**go/no-go 客观依据。**
- **G-B-1（usbipd 链路）**：attach 后 WSL 内 `/dev/ttyACM0` 出现且 agent 能打开（不报 permission/busy）；记录实际设备名。
- **G-T7（真机回环）**：契约第 2 节值回环 + 第 7 节 A1–A8 在**真机**复现（尤其 A2 延迟/A3 丢包在真串口下）。
- **波特率验收（并入 G-T7）**：921600 实链路稳定无大量 CRC/帧错（印证 A.3）。

### C.4 Phase B 现在能提前到什么程度
- **立刻可做**：B 节 usbipd 链路（G-B-1）——与固件进度无关。
- **T4/T-meta/T8 后**：真机起 micro-ROS session（G-T8-a）。
- **T5 后**：真机双向回环（G-T7，原 Phase B 终点）。
- 即**“Phase A 全闭环 → 才进 Phase B”的串行约束可打破**：usbipd 验证、F103 可行性 spike 可与 Phase A 收尾并行，尽早证伪/证实最大未知（20KB）。

### C.5 建议的下一步顺序（去风险优先）
1. **先解决当前阻塞**：~~修 loopback bug → A1–A8 全绿~~（已完成，见交接文档）→ G2/链路监控收尾。不受换芯影响（Int32 wire 不变）。
2. **并行两条线**：(a) 立即按 B 节打通 usbipd → WSL 见 `/dev/ttyACM0`（G-B-1）；(b) **T-meta + T8** 尽早回答“20KB 跑不跑得下”。
3. **T8 = go**：T4（F103 骨架）→ T5（transport + 双向）→ T7 真机回环（含 A1–A8 复现）。
4. **T8 = no-go**：触发 A.2 降级预案，**回来找用户拍板**，不在固件层硬扛。
5. 全程：契约修订（QoS history 取舍、board 名、备选串口），v1.1 → v1.2。

## 两个决策点（✅ 2026-06-19 已用户拍板）
1. **契约 QoS 取舍** ✅ **接受**：F103 侧 history 深度 10→1（WSL 侧保持 10），RELIABLE 不变。已落实到契约 v1.2（第 1 节 QoS 行 + 注解）。
2. **降级预案触发权** ✅ **确认**：若 T8 判定 20KB 跑不下 micro-ROS，**回来找用户**在“降功能 / 换轻量协议 / 换芯”间拍板，Tom/子 agent 不自行决定。
