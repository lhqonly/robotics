## 0. 一句话结论

换 F407 在 **RAM(192KB) / Flash(512KB-1MB) 上是从"紧箍咒"到"宽松"的质变**——T8 在 20KB F103 上都判 GO 且余 ~5.6KB，F407 几乎没有内存悬念。**真正的两个硬风险都在底层、且与 RAM 无关**：

- **闸门 A（ABI）**：F407 是 **Cortex-M4F、硬浮点（`-mfloat-abi=hard`+`-mfpu=fpv4-sp-d16`）**，libmicroros.a 必须用同一 float ABI + M4 **从头 `generate_lib` 重建**。ABI 不匹配 = 链接期 `multiple definition`/VFP register mismatch，或运行期诡异崩。这是开工第一闸门。
- **闸门 B（DMA 重做）**：F103 的 DMA 是 **DMA1 通道(channel)模型**;F407 是 **DMA2/DMA1 的 stream+channel 双层模型**，且 **USART1 的 DMA 请求在 DMA2(不是 DMA1)**。当前那段「CPAR/RCC 时钟屏障 + CNDTR 绕回」transport 是按 F1 DMA 推的，**不能照搬**——寄存器名(SxCR/SxPAR/SxM0AR/SxNDTR/LISR/HISR/LIFCR/HIFCR)、绕回判定、FIFO/直接模式、清标志序列全变。这是开工第二闸门，列为头号工程风险。

策略与 04 一脉相承:**先把两个底层闸门用最小 spike 各自证伪/证实,再叠 micro-ROS;先沿用 F103 的保守配置确保能跑,再逐步松绑省下来的 RAM,别一上来全放开引入新变量。**

> 与 04 (F401→F103) 的关键区别:那次最大风险是 RAM(往下挤、有压不进的真实可能);**这次 RAM 反而宽松,风险全在 ABI 重建 + F4 DMA 重写两个底层动作**。方向相反,别套用 04 的"省 RAM"思维定式。

## A. F103 → F407 差异矩阵 + 影响面（逐条:差异 → 改哪个文件）

> ⚠️ 凡标「待 Tom 查手册确认」的寄存器级事实(DMA stream 映射、PLL 具体分频、BRR 重算),**以 RM0090 / STM32F407 数据手册为准,不凭本文记忆定值**。本文给方向与已知约束,精确值由 Tom 在卡里查实。

### A.1 内核:Cortex-M3(无 FPU)→ Cortex-M4F(有 FPU)
| 维度 | F103(M3) | F407(M4F) | 改哪 |
|---|---|---|---|
| `-mcpu` | `cortex-m3` | **`cortex-m4`** | `toolchain-arm-m4.cmake`(新) |
| `-mfpu` | 不设 | **`fpv4-sp-d16`** | 同上 |
| `-mfloat-abi` | `soft` | **`hard`** | 同上 |
| FreeRTOS port | `portable/GCC/ARM_CM3` | **`portable/GCC/ARM_CM4F`** | `CMakeLists.txt` 源文件列表 + include 路径 |
| `configENABLE_FPU` | `0` | **`1`** | `FreeRTOSConfig.h` |
| startup/CMSIS device | `startup_stm32f103xb.s` / cmsis-device-f1 | **`startup_stm32f407xx.s` / cmsis-device-f4** | vendor 目录 + `CMakeLists.txt` + 宏 `STM32F103xB`→`STM32F407xx` |

- **硬约束(同 04 A.1 的镜像)**:三件套 `-mcpu/-mfpu/-mfloat-abi` 必须一起改且与 libmicroros 完全一致(见闸门 A)。F103 那张表是"砍 FPU",F407 这张是"开 FPU"——残留 `soft` 或漏 `-mfpu` 同样链接期爆。
- **FPU 上下文保存**:M4F 用 `ARM_CM4F` port,FreeRTOS 会在任务切换时保存/恢复 S0-S31。`configENABLE_FPU=1` 必开。若漏开仍用 CM3 port,浮点上下文不保存 → 多任务下浮点状态被踩,诡异错值(本项目当前 wire 不涉浮点,但 micro-ROS/未来控制律会用,必须从一开始就配对)。
- **浮点可用了**:F103 时代"固件路径禁浮点"是 M3 软浮点慢的妥协;F407 有硬 FPU,`clock_gettime` 的 64 位换算等仍建议走整数(纳秒精度、避免 FPU 异常),但不再是硬禁。**本换芯阶段不主动引入浮点逻辑**(控制 wire 仍 Int32/exo_msgs),保持变量最小。

### A.2 内存:20KB SRAM → 192KB(含 64KB CCM)/ 512KB–1MB Flash
- **事实(待 Tom 按具体型号数据手册确认)**:STM32F407 系列常见 **192KB SRAM = 112KB(SRAM1) + 16KB(SRAM2) + 64KB CCM**,Flash **512KB(VG)~1MB(ZG/IG)**。Nucleo-F407ZG 是 1MB Flash/192KB RAM(其中 CCM 64KB);若用 Discovery(STM32F407VG)是 512KB Flash。**具体板子/型号 → Flash/RAM 容量 → 链接脚本 MEMORY 段,以 Tom 核实的型号为准。**
- **影响面**:
  - 新链接脚本 `STM32F407xx_FLASH.ld`:`FLASH LENGTH`、`RAM LENGTH` 按型号填;`_estack` 仍指主 SRAM 顶。
  - **🔴 CCM RAM 不能放 DMA buffer(头号易错点)**:64KB CCM(Core-Coupled Memory,`0x10000000`)**只挂在 D-bus,DMA 控制器访问不到**。把 `rx_dma_buf`/`tx_dma_buf` 放进 CCM → DMA 静默读写失败(像极了 F103 的 CPAR=0 风暴,但根因完全不同)。**DMA buffer 必须留在 DMA 可达的 SRAM1/SRAM2,绝不进 CCM。** 链接脚本若划了 CCM 段,要显式把 DMA buffer 段锚到 SRAM。CCM 适合放任务栈/纯 CPU 数据。
- **配置松绑顺序(关键,别一上来全放开)**:
  1. **第一步:整段沿用 F103 的保守 `colcon.meta`**(pub/sub=1、STREAM_HISTORY=2、MTU=128、static pool、MAX_HISTORY=1)。先证明 F407 链路在已知好配置下能跑通——**把"换芯"这一个变量单独验证**,不和"松绑配置"混在一起。
  2. 链路 + RAM/ROM 量化绿了之后,**再逐步松绑**(候选:STREAM_HISTORY↑、MTU↑、MAX_HISTORY 1→10 让 F103 侧 depth 也回到契约的 10、任务栈放宽)。每松一项单独量化、单独回归,出问题能立即定位是哪一项。
  3. **契约张力可顺势消解**:04/v1.2 因 20KB 把 F103 侧 QoS depth 砍到 1(WSL 侧 10),是 RAM 取舍。F407 RAM 宽松后**可以把 MCU 侧 depth 提回 10、两端一致**——但这是**契约级变更(回退 v1.2 的分侧取舍)**,须 elon/用户在契约写明,不由 Tom 在松绑阶段私自改(见 Q3)。

### A.3 时钟树:F103 72MHz → F407 PLL 到目标主频
- **F103 现状(main.c)**:HSE 8MHz(bypass,来自 ST-Link MCO)→ PLL×9 = 72MHz;AHB/1=72M、APB1/2=36M、APB2/1=72M;Flash 2WS。USART1 挂 APB2(72MHz),BRR=0x4E。
- **F407 差异(方向,精确值待 Tom 查)**:
  - F407 主频可到 **168MHz**(典型)。PLL 结构不同:F4 用 `M/N/P/Q` 分频(`f_VCO = f_in/M*N`,`SYSCLK = f_VCO/P`),不是 F1 的单一 `PLLMULL`。寄存器是 `RCC->PLLCFGR`(M/N/P/Q + PLLSRC),与 F1 的 `RCC->CFGR.PLLMULL` 完全不同模型。
  - **HSE 源**:Nucleo-F407ZG 板载 8MHz(同样来自 ST-Link MCO,大概率仍 bypass);Discovery-F407VG 板载 8MHz 晶振(HSE 非 bypass)。**bypass 与否取决于具体板子**,F103 main.c 用了 `HSEBYP`,F407 要按实际板子定(待 Tom/用户确认板型)。
  - **APB 分频**:F407 APB1 最高 42MHz、APB2 最高 84MHz(@168MHz 主频时 APB1/4、APB2/2)。**USART 挂哪条总线、该总线时钟多少,直接决定 BRR**。F407 USART1 仍在 APB2。
  - **Flash 等待周期**:168MHz/3.3V 需 **5 WS**(F4 是 `FLASH->ACR` 的 `LATENCY`,数值不同于 F1 的 2WS),还要开 ART 加速器(prefetch/icache/dcache)。配错 = 取指错乱/硬错。
  - **🔴 BRR 必须按 F407 的 APB2 时钟重算**:921600 @ F407 APB2 时钟(若主频 168M → APB2=84M)的 BRR 与 F103(APB2=72M → BRR=0x4E)**不同**。且 **F4 USART 支持 8× 过采样(OVER8)**,16×/8× 选择影响 BRR 公式。**BRR 具体值待 Tom 按选定主频 + APB2 时钟 + 过采样模式手算并在卡里给出推导**(像 04 A.3 那张误差表一样,要给实际 baud 与误差 %)。
  - `SystemCoreClockUpdate()` 由 cmsis-device-f4 的 `system_stm32f4xx.c` 提供,按 F4 寄存器反算。

### A.4 外设 / HAL 寄存器模型:cmsis-device-f1 → cmsis-device-f4
- 本工程**不用 HAL,是寄存器级 + CMSIS Device 头**。换的是 CMSIS Device 包(f1→f4)与寄存器写法,不是 HAL API。
- **GPIO 模型质变**:F1 是 `CRL/CRH`(每脚 MODE[1:0]+CNF[1:0] 老模型 + AFIO 重映射);**F4 是现代模型**——`MODER`(每脚 2 位模式)、`OTYPER`(推挽/开漏)、`OSPEEDR`(速度)、`PUPDR`(上下拉)、**`AFRL/AFRH`(每脚 4 位 alternate function 选择,USART1 通常 AF7)**。`GPIO_Init()` 整段按 F4 重写;不再有 AFIO 重映射,改用 per-pin AF 寄存器。
- **USART 引脚 AF**:F103 USART1 = PA9/PA10(默认,无需 AF 选择);F407 USART1 仍可 PA9/PA10,但**必须在 AFRH 里把 PA9/PA10 设成 AF7(USART1)**。引脚号可沿用,但 AF 配置是新增动作。
- **RCC 使能位/总线**:F4 的外设时钟使能寄存器是 `RCC->AHB1ENR`(GPIO、DMA1/2 在 AHB1)、`RCC->APB2ENR`(USART1)等,位名与 F1 的 `APB2ENR/AHBENR` 不同。GPIO 在 F4 挂 **AHB1**(F1 挂 APB2)。
- **中断文件**:`stm32f1xx_it.c` → **`stm32f4xx_it.c`**。中断向量名变(`DMA1_Channel5_IRQHandler` → F4 的 `DMA2_Stream2_IRQHandler` 之类,取决于选定 stream);`#include "stm32f1xx.h"` → `"stm32f4xx.h"`。IDLE/ORE 清序列 F4 与 F1 也不同(F4 有独立的 `USART_SR` 行为差异 + 部分型号用 `ICR`,但 F407 仍是经典 `SR`+读 `DR` 清,待 Tom 核实)。

### A.5 🔴 DMA(头号风险,单列):F1 通道模型 → F4 stream+channel 模型
这是整个换芯**最大的重写量 + 最大的踩坑面**,必须当独立闸门(闸门 B)前置 spike,不能埋进大卡到最后才爆。

| 维度 | F103(DMA1 channel) | F407(DMA2 stream+channel) | 后果 |
|---|---|---|---|
| 控制器 | USART1 收发都在 **DMA1** | **USART1 收发在 DMA2**(不是 DMA1!) | 用错控制器 = DMA 完全不动 |
| 单元模型 | Channel(1-7),固定外设映射 | **Stream(0-7) + Channel(0-7) 双层**,Stream 选哪个外设由 Channel 决定 | 配置寄存器结构全变 |
| USART1_RX 映射 | DMA1 Ch5(固定) | **DMA2 Stream2/Ch4 或 Stream5/Ch4**(二选一,待 Tom 查 RM0090 DMA request mapping 表) | 选错 stream/channel = 收不到 |
| USART1_TX 映射 | DMA1 Ch4(固定) | **DMA2 Stream7/Ch4**(待 Tom 查表确认) | 同上 |
| 配置寄存器 | `CCR/CPAR/CMAR/CNDTR` | **`SxCR`(含 CHSEL[3:0] 选 channel)`/SxPAR/SxM0AR/SxM1AR(双缓冲)/SxNDTR/SxFCR(FIFO)`** | 整段 `DMA_Init` 重写 |
| 状态/清标志 | `DMA1->ISR / IFCR`(每通道几位) | **`LISR/HISR`(读)+ `LIFCR/HIFCR`(清)**,stream0-3 在 L、stream4-7 在 H,位偏移按 stream 算 | ISR 里清标志逻辑全变 |
| FIFO/直接模式 | 无 FIFO | **有 FIFO**(`SxFCR`),可直接模式(direct)或 FIFO 模式 | RX circular 建议直接模式起步,降复杂度 |
| 绕回当前位置 | `cur = SIZE - CNDTR`(+掩码) | `cur = SIZE - SxNDTR`(+掩码,逻辑同,寄存器名变) | `rx_dma_collect` 改寄存器名,绕回掩码逻辑可复用思路 |
| CPAR=0 时钟屏障坑 | 开 DMA1 时钟后回读 `RCC->AHBENR` | F4 同类坑仍在:**开 DMA2 时钟(`RCC->AHB1ENR` 的 DMA2EN)后须回读屏障**,否则首个 SxPAR 写可能被丢 | 同源坑,F4 上重新验证(开 `AHB1ENR` 屏障) |

- **circular + IDLE 收变长帧的机制可沿用**(F407 USART 也有 IDLE 中断),但**实现细节(stream 选择、FIFO 阈值、清标志、双缓冲 vs 单 circular)要在 F4 上重新推导**,不能 copy F103 的 `DMA_Init`/`rx_dma_collect`/`stm32f1xx_it.c`。
- **CPAR=0 同源坑要主动复验**:F103 那个"开时钟后头一两个外设写被静默丢 → CPAR=0 → 0x00 风暴"的坑,在 F4 上对应"开 `AHB1ENR.DMA2EN` 后立即写 `SxPAR` 可能被丢"。**闸门 B 的 spike 必须显式带上 `(void)RCC->AHB1ENR;` 回读屏障并验证 SxPAR 写入生效**(读回 SxPAR 确认非 0)。这是已知坑,提前堵。

### A.6 烧录 / 物理链路
- **ST-Link SWD 烧录不变**:`st-flash --connect-under-reset write build/f407-microros.bin 0x08000000`(WSL+usbip 下仍须 `--connect-under-reset`)。地址仍 `0x08000000`。
- **通信口物理链路**:沿用 v1.3 的**独立 USB-TTL + USART1(PA9=TX/PA10=RX)**,`/dev/ttyUSB0`,921600 8N1。**引脚号可沿用 PA9/PA10**(F407 同样支持,但要配 AF7,见 A.4)。usbipd attach 两设备流程(04 B 节)不变。
- **板型确认**:F103 是 Nucleo-F103RB。F407 用 **Nucleo-F407ZG** 还是 **Discovery STM32F407VG**?二者 Flash 容量(1MB vs 512KB)、HSE(bypass vs 晶振)、引脚排布、ST-Link 版本不同 → 影响链接脚本 + 时钟树 + 烧录。**待用户确认板型(Q1)**。

## B. 两个必须前置的硬风险闸门（开工第一/第二动作）

> 节奏同 04:把项目级最大未知做成**前置去风险 spike**,先回答 go/no-go,别埋进大卡到最后才爆。两个闸门可并行起步,但都必须在「叠 micro-ROS 完整应用」之前绿。

### 🔴 闸门 A — libmicroros 必须 hard-float + M4 从头重建（开工第一动作）
- **为什么是闸门**:F407 用 `-mfloat-abi=hard`。当前 F103 的 `libmicroros.a` 是 `cortex-m3 + soft-float` 编的,**ABI 完全不兼容**。直接拿去链 F407 固件 → 链接期 `error: ... uses VFP register arguments, ... does not` / `multiple definition`,或更坏:链过了但运行期参数传递错乱诡异崩。
- **怎么做**(复用 T5 经验,改 M4/hard):
  1. 写新的 `toolchain-arm-m4-libmicroros.cmake`:`include` 新固件 toolchain `toolchain-arm-m4.cmake`(M4F/hard 唯一真源),再追加 T5 那两个裸机移植宏(`-DCLOCK_MONOTONIC=1 -DCLOCK_REALTIME=0`,见 F103 同名文件)。
  2. 在 micro_ros_setup 工作区(`~/uros_ws/firmware/...`)用这个 toolchain 跑 `generate_lib`/`build_firmware.sh`,产出 **M4F/hard 的 `libmicroros.a` + include/**。
  3. **沿用 F103 的 `colcon.meta`**(保守配置,先不松绑),把它喂进重建(注入点/流程同 04 与交接文档 T8)。
  4. 拷进 `firmware/f407-microros/ThirdParty/microros/`。
- **验收(去风险判据,先于一切 micro-ROS 应用)**:
  - `arm-none-eabi-readelf -A libmicroros.a`(或对其中一个 .o)确认 **Tag_FP_arch = VFPv4-SP、Tag_ABI_VFP_args = VFP registers**(即 hard-float、M4),无 soft-float 残留。
  - 一个**最小 spike**(只 `rclc_support_init` + 1 pub,甚至只链接不调用)能**链接通过**(无 multiple definition / VFP mismatch),证明 ABI 配对。**编译过 ≠ 赢,要先确认 readelf 属性 + 链接期无 ABI 错。**
- **失败处置**:ABI 报错先核对三件套 flag 在「固件 toolchain」与「libmicroros toolchain」是否字节一致(T5 的教训:两处 flag 必须同源);确认 micro_ros_setup 没有用自己默认的 soft-float toolchain 覆盖。

### 🔴 闸门 B — F4 DMA2 USART RX/TX transport 最小 echo spike（开工第二动作）
- **为什么是闸门**:A.5 的 DMA 重写量 + 踩坑面最大。**先用最小裸机 echo(不带 micro-ROS)在 F407 DMA2 上验证逐字节双向**,复刻 F103 当年"先 echo 全值域全绿再叠 micro-ROS"的去风险路径(交接文档:9600/115200/921600 全值域逐字节 echo 是 micro-ROS 之前的硬门)。
- **spike 内容(最小,纯物理层)**:
  1. 时钟树到目标主频(A.3)+ GPIO PA9/PA10 配 AF7(A.4)+ USART1 BRR(重算)。
  2. **DMA2 RX circular(选定 stream/channel)+ USART1 IDLE 中断 + DMA2 TX**,把收到的字节原样回显。
  3. **显式带 `(void)RCC->AHB1ENR;` 时钟屏障**(A.5 同源坑),并读回 `SxPAR` 断言非 0。
  4. 主机侧发 `0x00..0xFF` 全值域 + 256 字节连续,验证逐字节精确回显(F103 的硬门标准)。
- **验收(去风险判据)**:
  - 921600 下 host→F407→host 逐字节精确双向 echo,**含 `0x00..0xFF` 全值域**,无 0x00 风暴/无丢字节/无字节爆炸。
  - 故意制造 RX 持续流压测 circular 绕回(`SxNDTR` 绕回 + IDLE 双触发路径)不丢、不死循环(F103 的 CNDTR 绕回死循环坑要在 F4 上确认不复现)。
- **失败处置**:0x00 风暴 → 查 SxPAR 是否被时钟屏障漏掉而为 0、是否 DMA buffer 误放进 CCM(A.2 坑)、stream/channel 是否选错;字节爆炸 → 查 circular 绕回掩码 + IDLE/ORE 清序列(F4 序列待核实)。

> **两闸门关系**:A 与 B 互相独立可并行(A 是构建链/ABI,B 是裸机外设)。**两者都绿后**才进 M1 骨架整合 → micro-ROS 建链。任一闸门红 = 停在该闸门排查,不往上叠,避免「ABI/DMA 底层未定就堆 micro-ROS」把根因埋深。

## E. F407 换芯 与 exo_msgs M-B 的先后关系（elon 建议）

**结论:F407 骨架 + 建链(F4-0~F4-2)先打通,M-B 固件实现直接在 F407 上做,不在 F103 上做 M-B。** 与用户倾向一致,理由如下(独立判断):

1. **避免重复劳动(核心理由)**:M-B 的固件改动集中在 `microros_app.c`(收 ExoCmd 解包/回填 ExoStatus、DWT stamp、CRC)+ 重建 libmicroros(喂 exo_msgs)。若先在 F103 做完 M-B,**换 F407 时这两块都要重做**:libmicroros 必须为 M4F/hard **再重建一次**(且这次还要带 exo_msgs),`microros_app.c` 虽逻辑可移植但 **DWT CYCCNT@72MHz 的 stamp 换算要改成 F407 主频**(H3 的 64 位回绕高频源周期/换算系数都随主频变)。在 F103 上做 = 做两遍。
2. **DWT stamp 与主频强耦合**:M-B 的 H3(DWT CYCCNT 64 位回绕)写死了 **@72MHz、59.65s 回绕**。F407 主频不同(如 168MHz → CYCCNT 回绕 ~25.6s),回绕扩展的高频源周期约束、`ns = cycles*1e9/f` 的 f 都变。**在 F407 上一次做对,比 F103 做完再改对省事且少一次出错机会。**
3. **B1 流缓冲闸门只需测一次**:M-B 的第一闸门(任务 5.0,exo_msgs bin create 描述消息能否塞进 STREAM_HISTORY=2/256B)在 F407 上测即可——而且 **F407 RAM 宽松,即便塞不下也能直接调大 STREAM_HISTORY,不像 F103 要在 20KB 里抠**。F407 反而让 B1 的处置空间更大。
4. **依赖顺序**:M-B 的前置是"重建 libmicroros(喂 exo_msgs)"。F407 的闸门 A 本来就要重建 libmicroros。**把"重建"这个动作合并**:F4-A 重建时若已确定走 F407 路线,可直接喂 exo_msgs(或先 Int32 跑通骨架、F4-A' 再喂 exo_msgs 重建)——见 Q4。

**但有一个前置约束(诚实标注风险)**:上述建议成立的前提是 **F407 闸门 A/B 能绿**(ABI 重建 + DMA 重写成功)。若闸门 B(F4 DMA)卡住时间过长,而 M-B 又有交付压力,则退路是:**先用现成的 `int32-baseline`/F103 链路把 M-B 在 F103 上跑通拿到验证(用现有 09/10 卡),F407 并行去风险**——这是"不把 M-B 押在 F407 单点"的保险。**默认走"M-B 直接在 F407"**,但把"F407 闸门久攻不下则 M-B 回退 F103"写成显式 plan B。这点请用户在 Q4 一并拍板。

**对 09/10/11 卡的影响**:若采纳"M-B 在 F407 做",则 09/10 卡的"重建 libmicroros(复用 toolchain.cmake)/microros_app.c 路径/DWT@72MHz"等措辞要更新为 F407(M4 toolchain、f407 路径、DWT@选定主频)。11 卡的 B1/H1/H2/H3 四条硬约束**全部仍适用**(芯片无关的契约约束),只是 H3 的频率参数随 F407 变。**这属于 M-B 卡的增量修订,待 Q4 拍板后由 elon 出**(不在本文最终任务卡范围,先标注)。
