### ✅ 已解决：USART RX 故障 = 两个 DMA RX 固件 bug（2026-06-20 真机查清并修复）
**现象（历史）**：MCU TX 完美，但 host→MCU 方向，发任何字节（0x55/0xFF/'A'）→ MCU 都读成 **0x00 + 字节爆炸**（发 4 收 ~226），9600/115200/921600 各速率都一样坏。
**真实根因（已确诊，不是硬件/PA10/信号完整性问题）**：两个独立的 DMA RX 固件 bug。
1. **主因（CPAR 被静默丢成 0）**：`DMA_Init()` 里 `RCC->AHBENR |= RCC_AHBENR_DMA1EN;` 开 DMA 时钟后**紧跟着写 DMA 寄存器**，开时钟后头一两个外设写被 STM32 静默丢弃 → DMA1 Ch5（RX）的 `CPAR`（外设地址）被丢成 `0` → DMA 从地址 0 狂读而非读 `USART1->DR` → RXNE 永不清 → 第一个字节一到，DMA 自由奔跑无限搬 `0x00`。**这正是「TX(Ch4) 完美、RX(Ch5) 坏」那个诡异不对称的真正原因**：TX 的 CPAR 写得晚、时钟已稳，所以一直正常。**修复**：开时钟后加 `(void)RCC->AHBENR;` 回读屏障。
2. **次因（CNDTR 绕回死循环）**：`rx_dma_collect()` 的 `cur = SIZE - CNDTR` 在 circular 绕回瞬间能取到 256，而索引被 `&255` 掩在 `[0,255]` 永不相等 → while 死循环。**修复**：cur 也加掩码。
**已验证**：修复后 **9600 / 115200 / 921600** 全部逐字节精确双向 echo（含 `0x00..0xFF` 全值域、256 字节连续）。
**纠正之前的错误归因（留痕）**：此前曾把故障怀疑成「PA10 引脚硅损伤 / 高波特率信号完整性不足」，并据此把契约降到 115200（v1.4）。**该归因是错的**（基于一份错误的现场诊断），与波特率、PA10、信号完整性均无关。真因是上述 DMA bug，已修复。波特率**定回 921600**（原始设计目标，实测稳定），契约已 bump 到 **v1.5**。

## 重要环境约束(踩过的坑)
- **sudo 装包必须用户在交互终端手动跑**:agent 会话无 TTY 输不了 sudo 密码,所有 `apt` 安装都由用户执行脚本完成。
- **子 agent 的 Bash 被沙箱大幅收紧**:实测连含 `source` 的命令都被拦(即便配置已 allow);但 **主 agent 的 Bash 可放行 `colcon`/`ros2`/`source`** —— 所以"build/launch/实跑验证"这类执行交给主 agent 做,子 agent(Tom)专注写代码落盘。这是本项目稳定的分工模式。
- env-check.sh / 涉及 ROS 的脚本 source 前需 `set +u`(ROS setup.bash 引用未初始化变量,与 set -u 冲突)。
- `ros2 topic info -v` 的 History(Depth) 恒显 UNKNOWN(rmw 发现不传 depth),非 QoS 缺陷;以代码里的 QoSProfile 为准。
- 同一 WSL 上跑多份 launch 会在默认 DDS 图里串台(topic 同名互联),验证时确保只有一份实例。
