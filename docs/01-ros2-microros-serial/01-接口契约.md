> **v1.5 变更说明**：串口基线波特率**定回 921600**（BRR=0x4E），推翻 v1.4 的 921600→115200 降速决策。8N1/DMA/双向 不变；wire 与回调契约不变。
> 决策依据（2026-06-20 真机根因查清并修复）：
> ① **v1.4 的降速归因是误判**（基于当时一份错误的现场诊断），现已被本次真机定位推翻。RX「收到全 `0x00` + 字节爆炸」**不是硬件/信号完整性/PA10 引脚问题，而是两个 DMA RX 固件 bug**：
>   - **主因（CPAR 被静默丢成 0）**：固件 `DMA_Init()` 中 `RCC->AHBENR |= RCC_AHBENR_DMA1EN;` 开 DMA 时钟后**紧跟着写 DMA 寄存器**，开时钟后头一两个外设写会被 STM32 静默丢弃 → DMA1 Ch5（RX）的 `CPAR`（外设地址）被丢成 `0` → DMA 从地址 0 狂读而非读 `USART1->DR` → RXNE 永不清 → 第一个字节一到，DMA 就自由奔跑无限搬 `0x00`。这正是「**TX(Ch4) 完美、RX(Ch5) 坏**」那个诡异不对称的真正原因：TX 的 CPAR 写得晚、时钟已稳，所以一直正常。**修复**：开时钟后加一行 `(void)RCC->AHBENR;` 回读屏障。
>   - **次因（CNDTR 绕回死循环）**：`rx_dma_collect()` 的 `cur = SIZE - CNDTR` 在 circular 绕回瞬间能取到 256，而索引被 `&255` 掩在 `[0,255]` 永不相等 → while 死循环。**修复**：cur 也加掩码。
> ② **已验证**：修复后 **9600 / 115200 / 921600** 全部逐字节精确双向 echo（含 `0x00..0xFF` 全值域、256 字节连续）。921600 实测好用，不存在「上升时间不够 / 信号完整性」问题——v1.4 的该项担忧不成立。
> ③ **波特率基线定回 921600**（原始设计目标）：终局是实时电机控制，链路延迟敏感，波特率越高越好；921600 既然实测稳定，就没有理由降速。固件 BRR 已改回 `0x4E`。
> **历史误判留痕**：v1.4 曾把基线从 921600 降到 115200，理由是「真机 RX 故障与波特率无关、PA10 疑似硅损伤、为避信号完整性风险而降速、115200 作为长期工程选择」。该归因基于错误诊断，已被 v1.5 全面推翻——RX 故障是 DMA CPAR/CNDTR 固件 bug，与波特率、PA10、信号完整性均无关，且已修复。
> **v1.3 变更说明**：通信口从板载 ST-Link **VCP(USART2 PA2/PA3)** 改为**独立外接 USB-TTL 上的 USART1(PA9=TX/PA10=RX)**；ST-Link 仅保留 SWD 烧录。原因：VCP 与 SWD 共用同一 ST-Link/USB，经 usbip 转发 + 老 ST-Link 固件(V2J31) 导致 host→MCU(RX) 收不到真实数据（主 agent 真机实测：TX 完美、RX 收不到）。改独立 UART 隔离烧录与通信，去掉 VCP/usbip/烧录扰动所有混淆因素。WSL 下通信口 = `/dev/ttyUSB0`，ST-Link = `/dev/ttyACM0`(仅烧录)，两者都需 usbipd attach。USART1 在 APB2(72MHz)，**BRR=0x4E（921600，v1.5 定回；v1.4 曾误降为 0x271=115200）**，DMA1 Ch4/5。
> **v1.2 变更说明**：因 MCU 从 STM32F401 换成 **Nucleo-F103RB（仅 20KB SRAM）**，落实两点契约级调整（决策已经用户拍板，详见 `04-变更-F103换芯与硬件到位.md`）：
> ① **QoS 历史深度按方向分侧**：F103（micro-ROS client）侧 History Depth = **1**（`RMW_UXRCE_MAX_HISTORY=1`，省 RAM），WSL（ROS2）侧维持 Depth = **10**；RELIABLE 两端不变。代价（MCU 侧缓存浅、抖动更易丢）由第 7 节监控**如实暴露**，不掩盖，故不违背安全哲学。
> ② 串口物理层 board 名 F401RE→F103RB、备选独立串口 USART6→**USART3（PC10/PC11）**；USART2=PA2/PA3 经 VCP 不变。
> **v1.1 变更说明（保留）**：新增「第 7 节 链路健康监控」，把回环监控的要求从「值对得上即可」升级为**实时控制环标准**——
> 终极链路(SoC ROS2 ↔ STM32 micro-ROS 控关节电机)对延迟/丢包**零容忍**(安全关键)，监控本身必须能**如实暴露**延迟与丢包，**绝不允许静默掩盖**。
> 本次同时落实 G2 三个遗留项的契约级处置：P1-1(禁止静默驱逐)、P1-2(重复 echo 语义)、P1-3(计数器回绕语义)。
> wire 格式不变(最小闭环仍用 `std_msgs/Int32`),要求全部可在 Phase A 无硬件、用 loopback 注入延迟/丢弃来验证。

## 1. Topic 契约

### 1.1 `/exo/cmd_heartbeat`（WSL → MCU）
| 项 | 约定 |
|----|------|
| 类型 | **`exo_msgs/msg/ExoCmd`**（v1.7；原 `std_msgs/msg/Int32`） |
| 发布者 | WSL 侧 `exo_cmd` 节点 |
| 订阅者 | MCU micro-ROS 应用 |
| 频率 | 10 Hz（周期 100 ms） |
| 语义 | `header.seq` = 序列号计数器，mod `2^32` 回绕（§7.6）；`header.stamp_mono_ns` = 发送方单调时钟纳秒（§7.1）；`header.crc` = 应用级 CRC（默认 0，§7.9）；`payload` = 心跳/回环值，**与 `header.seq` 解耦** |
| QoS | Reliability=**RELIABLE**, History=**KEEP_LAST**；Depth：WSL 侧=**10**，F103 micro-ROS 侧=**1**（v1.2，20KB RAM） |

> **v1.7 字段**（`ExoCmd` = `ExoHeader header` + `int32 payload`；`ExoHeader` = `uint32 seq` + `uint64 stamp_mono_ns` + `uint32 crc`）：把 seq / 时间戳 / 完整性自检从「WSL 侧推断」升格为 wire 显式字段。`payload` 与 `header.seq` **解耦**——配对只看 `header.seq`，payload 任意取值不影响回环判定（exo_msgs 相对 Int32 的核心结构改进）。

### 1.2 `/exo/mcu_status`（MCU → WSL）
| 项 | 约定 |
|----|------|
| 类型 | **`exo_msgs/msg/ExoStatus`**（v1.7；原 `std_msgs/msg/Int32`） |
| 发布者 | MCU micro-ROS 应用 |
| 订阅者 | WSL 侧 `exo_cmd` 节点 |
| 频率 | ~10 Hz（响应式：每收到一条 cmd_heartbeat 即回发一条） |
| 语义 | `header.seq` = **原样回填**收到的 `cmd.header.seq`；`header.stamp_mono_ns` = MCU 用**自己单调时钟重盖**（不回填 cmd 的 stamp）；`header.crc` = 回填方按规范重算（开关开时，§7.9）；`payload` = **原样回填** `cmd.payload`（回环校验） |
| QoS | Reliability=**RELIABLE**, History=**KEEP_LAST**；Depth：WSL 侧=**10**，F103 micro-ROS 侧=**1**（v1.2，20KB RAM） |

> **v1.7 回填语义**（`ExoStatus` 结构同 `ExoCmd`，语义=MCU 回填）：`header.seq` 精确回填以供 WSL 侧配对（配对用精确相等，不看 payload）；`stamp_mono_ns` 由回填方重新生成（用于 MCU 侧相对时效，不参与整链 RTT）；`payload` 原样回填供值正确性校验。

> QoS 注意：micro-ROS 端默认 Best-Effort 较省资源；本契约要求 **RELIABLE**（两端一致，否则匹配失败）以便回环判定确定性——Tom 在固件侧需显式设置 reliable QoS。
> **v1.2 历史深度分侧（已用户拍板）**：F103RB 仅 20KB SRAM，Depth=10 的历史 + RELIABLE 重传缓存是 RAM 杀手。故 F103 micro-ROS client 侧 `RMW_UXRCE_MAX_HISTORY=1`（深度 1），WSL（ROS2）侧维持 Depth=10。RELIABLE 语义不变。代价是 MCU 侧缓存浅、链路抖动时更易丢——但这**应被第 7 节监控如实暴露（计数/告警），绝不静默掩盖**，符合零容忍安全哲学。理由与降级预案详见 `04-变更-F103换芯与硬件到位.md` A.2。

## 2. 回环正确性判据（验收基准）
WSL 侧记录每条发出的 `cmd_heartbeat.data` = N，期望在有限延迟内收到 `mcu_status.data` = N。
- **通过**：连续收到的 status 值与先前发出的 heartbeat 值序列一致（允许固定的链路延迟/相位差，不允许丢序或错值）。
- **失败**：值不匹配、长时间无回（>1 s）、或序列错乱。

> 注：第 2 节是「值正确性」的基线判据。v1.1 起，**链路健康(延迟/丢包)的可观测性要求见第 7 节**，二者一并构成验收基准。

## 4. Agent 启动契约
```bash
ros2 run micro_ros_agent micro_ros_agent serial --dev /dev/ttyUSB0 -b 921600 -v6
```
- **v1.3：`--dev` 是 USB-TTL 的 `/dev/ttyUSB0`**（通信口），不是 ST-Link 的 ttyACM0（那只烧录）。以实际枚举为准。
- **v1.5：`-b` 基线为 921600**（BRR=0x4E）。agent 波特率必须与固件 USART BRR 一致，否则建链失败。（v1.4 曾误用 115200/0x271，已推翻。）
- agent 必须在 MCU 复位/上线前或同时运行；建议固件侧带重连逻辑。

## 6. 演进约定
- 最小闭环原用 `std_msgs/msg/Int32`。闭环稳定（Gill G-series 验收通过）后，引入自定义 `exo_msgs`（含时间戳、序列号、CRC 等）。**v1.7：exo_msgs 阶段已落地（M-A，WSL 侧先行）**——WSL 侧 `colcon build` + `colcon test` 全绿、loopback 复现 A1–A8、`/exo/link_health` 发布；**固件侧待真机（M-B）**：F103 收 `ExoCmd` 解包 / 回填 `ExoStatus`、重建 libmicroros 喂入 `exo_msgs` type support、MCU 用 DWT CYCCNT @72MHz 折算 `stamp_mono_ns`、CRC 端到端字节序对齐。M-A 与 M-B 解耦：M-A 全绿后真机仍可继续跑已验证的 Int32 链路（git tag `int32-baseline` 退路）。

- 第 7 节定义的链路健康监控**要求(行为/可观测性)**在 v1.1 即生效;其中需要 wire 改动才能更精确实现的部分(如把发送时间戳放进消息、扩宽序列号)在 `exo_msgs` 阶段落实——届时本节会指明哪些第 7 节要求由 wire 直接保证、哪些仍由 WSL 侧推断。
- **波特率（v1.5）**：基线 921600 已是原始设计目标且真机实测稳定，作为长期基线。固件已验证 9600/115200/921600 全部精确双向 echo，若未来某些场景需要其它速率，物理层已无障碍，按需评估即可，但默认维持 921600。
- 当前契约版本：**v1.7**（v1.0 = 最小闭环回环；v1.1 = 链路健康监控；v1.2 = F103RB 换芯/QoS 分侧/usbipd；v1.3 = 通信口改独立 USB-TTL 上的 USART1(PA9/PA10)、ST-Link 仅烧录、通信口 /dev/ttyUSB0；v1.4 = 波特率基线 921600→115200，**已被 v1.5 推翻（基于错误的 RX 故障归因）**；v1.5 = 波特率基线**定回 921600**，BRR=0x4E，真实根因为已修复的 DMA RX CPAR/CNDTR 固件 bug，与波特率无关；v1.6 = §7.5 重传/错值三类判定澄清 + Gill 发现 #1；v1.7 = **exo_msgs M-A 落地（WSL 侧先行）**：Int32→ExoCmd/ExoStatus、seq/stamp/crc 升格为 wire 字段、`/exo/link_health` 落地、seq 模数 2^31→2^32、新增 §7.9 应用级 CRC）。

## 7. 链路健康监控（v1.1 新增，安全关键）

### 7.0 为什么需要这一节（第一性原理）
终极链路是 **SoC(ROS2) ↔ STM32(micro-ROS) ↔ 关节电机**。电机控制环对**端到端延迟**和**命令完整性**零容忍：
- 一条迟到的命令 = 关节在错误时刻动作 → 物理风险；
- 一条丢失的命令 = 关节状态与上位机预期失步 → 物理风险;
- **最危险的不是延迟/丢包本身,而是监控把它们"悄悄吃掉"**——让操作者以为链路健康。

因此监控的设计目标不是「值对得上就放行」,而是:**链路的每一次延迟超限、每一次丢包,都必须被如实测量、计数、上报,任何条目的消失都必须有据可查。** 这直接否定了 v1.0 回环监控里 `min()` 静默驱逐的做法(P1-1)。

本节所有要求在 **Phase A 无硬件**即可验证:loopback 节点可被人为注入「延迟」「丢弃」「重复」来制造异常,监控必须如实报警/计数。

### 7.1 延迟可测（RTT 测量）— 要求 M1
- WSL 侧 `exo_cmd` **每发出一条 `cmd_heartbeat`(值 = N)**,必须记录该值的**发送时刻** `t_send[N]`(单调时钟,如 ROS clock 的 steady time / `time.monotonic()`,**不可**用 wall clock,避免 NTP 跳变污染)。
- **收到 `mcu_status`(值 = N)** 时,计算往返延迟 `RTT[N] = t_recv - t_send[N]`。
- RTT 必须被**记录并可观测**,至少满足其一(Phase A 接受 (a)):
  - (a) **结构化日志**:每条 matched echo 打一行,含 `seq=N`、`rtt_ms`;
  - (b) 滚动统计周期性打印:窗口内 RTT 的 `count / min / mean / p95 / max`;
  - (c) (演进项,exo_msgs 阶段)发布到诊断 topic `/exo/link_health`(见 7.7)。
- **wire 边界说明（v1.7 升格）**:exo_msgs 阶段起,`t_send[N]` 从「WSL 侧本地配对的易失状态」**升格为 wire 上的 `header.stamp_mono_ns`**(发送方单调纳秒,禁 wall clock)。但 **RTT 仍是权威延迟指标**:由 **WSL 侧本地配对** `t_send[N]`/`t_recv` 得到「WSL→Agent→MCU→Agent→WSL」整链 RTT。**单程延迟不作为权威指标**——缺 MCU↔WSL 时钟同步(§7.1 时钟不可比),`header.stamp_mono_ns` 仅用于①记录(bag/诊断)②MCU 侧相对时效地基,不据它算跨端单程。回填方(MCU)用**自己的单调时钟重盖** `stamp_mono_ns`,不回填 cmd 的 stamp(两端时钟不可比)。
- **实现状态(v1.7)**:本节 wire 行为 **M-A WSL 侧已实现**;**固件侧 wire 行为(MCU 用 DWT CYCCNT@72MHz 重盖 stamp、64 位回绕扩展)待 M-B 真机验证**(见任务卡 09/10、放行复核 11 之 H3)。

### 7.2 超限可报（延迟阈值告警）— 要求 M2
- 定义可配置阈值 `rtt_warn_ms`(软阈值,告警)与 `rtt_deadline_ms`(硬阈值,见 7.3 丢包判据)。
- **`RTT[N] > rtt_warn_ms` 时必须告警**(WARN 级日志或诊断 topic),内容含 `seq=N`、实测 `rtt_ms`、阈值。
- **Phase A 占位阈值(明确为占位,非最终值)**:
  | 参数 | Phase A 占位值 | 含义 |
  |------|--------------|------|
  | `rtt_warn_ms` | **50 ms** | 软告警;约为 10 Hz 心跳半个周期,留裕度 |
  | `rtt_deadline_ms` | **200 ms** | 硬死线;超过即判丢包(见 7.3),约 2 个心跳周期 |
- **这些值是占位,不是工程定论。** 真实阈值**由关节电机控制环的实时性需求决定**(取决于控制环频率、机械带宽、安全裕度),必须做成**运行时可配置参数**(ROS param / 配置文件),不得硬编码。当控制环频率从 10 Hz 提升到电机实际需要的频率(如 500 Hz–1 kHz)时,阈值将随之收紧到毫秒/亚毫秒级。
- 约束:`rtt_warn_ms < rtt_deadline_ms`;两者都应与心跳周期解耦(用绝对时间,不用「周期数」硬编)。

### 7.3 丢包可检（deadline 判据 + 计数）— 要求 M3
- **丢包判据(明确定义)**:某个已发出的值 N,在其 `t_send[N] + rtt_deadline_ms` 之前**仍未收到对应 echo**,即判定为**丢失**。
- 丢失必须:
  1. **显式计入丢包计数器** `lost_count`(单调累加,见 7.6 回绕语义);
  2. 打 **ERROR/WARN 级**日志,含 `seq=N`、`waited_ms`、判据阈值;
  3. 该条目从 in-flight 结构中移除(因已结算为「丢失」,非静默丢弃)。
- **派生指标**:维护 `sent_count`、`matched_count`、`lost_count`,使 `丢包率 = lost_count / sent_count` 可被计算与观测。
- **零容忍语义**:在终极链路上,`lost_count` 的任何增长都是安全事件——本契约不在监控层「容忍」丢包,只要求**如实暴露**,是否触发安全动作(降级/急停)由上层安全监控决定(后续需求)。

### 7.4 禁止静默驱逐（in-flight 结构约束）— 要求 M4（直接修复 P1-1）
- in-flight(已发出未确认)集合**必须是基于 deadline/超时的有界结构**。**严禁**用 v1.0 的 `min()` 静默驱逐(P1-1):那会在链路退化时把迟到/积压的条目悄悄删掉,正是「掩盖延迟/丢包」的反模式。
- 具体约束:
  - 每个 in-flight 条目带 `t_send[N]`(或等价 deadline `t_send[N] + rtt_deadline_ms`);
  - **唯一允许移除条目的两种途径**:① 收到对应 echo(结算为 matched,记 RTT)→ M1;② 超过 deadline(结算为 lost,`lost_count++` 并告警)→ M3。
  - **不存在第三种「为了限容量而丢掉最老条目」的无声路径。** 若需限制内存上界,达到上界时**也必须把被挤出的条目结算为 lost 并告警**,不得无声丢弃。
- 结果:in-flight 结构的大小本身成为一个健康信号——它持续增长意味着 echo 跟不上发送(积压),这正是要暴露的危险,不能被驱逐逻辑抹平。

### 7.5 重复 / 重传 echo 语义（P1-2）— 要求 M5
- **v1.7 seq 来源升格**:seq 不再由 `payload` 整数兼作,而是 wire 上的显式字段 **`header.seq`**(`uint32`),与 `payload` 解耦。WSL 侧配对、三类判定(`duplicate` / `stale_duplicate` / `UNMATCHED`)语义**不变**,仅 seq 取自 `header.seq`、越界域从 `[0,2^31)` 改为 **`[0,2^32)`**(随 §7.6 模数升级)。payload 任意取值不影响配对——配对只看 `header.seq`,这是 exo_msgs 相对 Int32 的核心结构改进,必须有测试守住。
- **实现状态(v1.7)**:WSL 侧配对/三类判定 **M-A 已实现**;**固件侧 wire 行为(seq 原样回填、payload bit-exact 透传不做任何变换)待 M-B 真机验证**(见任务卡 09/10、放行复核 11 之 H1)。
- 背景:RELIABLE QoS 下,DDS/Agent 重传可能导致**同一个值 N 的 echo 被收到多次**。v1.0 把第二次及以后的 echo 落入 else 分支误报 UNMATCHED(P1-2)。
- v1.1 语义:
  - **首次**收到 N 的 echo:正常 matched,记 RTT,从 in-flight 移除;
  - **再次**收到已结算的 N 的 echo:判定为**重复 echo**,这是 RELIABLE 重传的**正常现象**,**不得报成 UNMATCHED/错值告警**;
  - 但重复**必须可观测**:维护 `duplicate_count` 计数器并(至少 DEBUG/INFO 级)记录。重复率异常升高本身是链路质量信号(说明丢/重传频繁)。
- **v1.6 澄清(消除原 v1.1 歧义 + 落实 Gill 发现 #1;owner 可推翻第②条边界策略)**:原文「从未发出过**或已超出合理窗口**的值才是 UNMATCHED」与「曾经发出过的值不算错误」**自相矛盾**。现按下述裁决统一,判定分三类:
  - ① **曾发出、仍在 `settled_window` 内**的重传 → `duplicate`(DEBUG,计 `duplicate_count`),同 v1.1;
  - ② **曾发出、但已挤出 `settled_window`** 的迟到重传 → **`stale_duplicate`(不是 UNMATCHED)**:契约取「曾经发出过的值不算错误」为准,`settled_window` 仅决定 `duplicate` vs `stale_duplicate` 的**标签**,不决定「重复 vs 错误」。**判据 = wrap-safe「曾发出」**(`forward_distance(seq, _next_seq) ∈ [1, min(sent_count, 2^32-1)]`,v1.7 模数 2^32),而非 `settled_window` 成员。**独立计数 `stale_duplicate_count`(不并入 `duplicate_count`)+ 至少 INFO 级**,因为它专门指示「`settled_window` 配小了/链路积压/迟到严重」,是与普通 dup 不同性质的信号,不得静默(呼应 7.5「重复必须可观测」与安全哲学)。
    - *owner 待裁(降级 Medium)*:此策略把「板子重放一个**曾发过的**带内旧值」也判 benign,无法与合法重传区分。Gill/Codex 评估认为可接受(重放历史值危害远低于回全新乱值),但若 owner 要求更严,可把②再细分(如只放过 `[deadline×频率×裕度]` 窗口内的重传、之外仍报)。
  - ③ **从未发出过的值**,或**任何落在 `[0, 2^32)` 发送域之外的 echo(负数 / ≥2^32)** → **UNMATCHED(错误)**。后者尤为关键:发送侧只产生 `[0,2^32)`(v1.7 模数),越界值必为板子故障/帧损坏/注入,**必须在取模归一前就判 UNMATCHED**,否则会被 wrap 取模 alias 进「曾发出」带而漏报(Gill 发现的 High 漏洞,已修)。漏报比误报危险。

### 7.6 计数器回绕 / 取模语义（P1-3）— v1.7 落实为 wire `header.seq`（uint32, mod 2^32）
- 背景:Int32 baseline 下 `cmd_heartbeat.data` 是 `Int32`,模数 `2^31`,10 Hz 下约 **6.8 年**回绕;v1.0 的 `value > last_status` 等比较在回绕点会误判(P1-3,Codex 标 High/远期)。**v1.7 起 seq = wire 上的 `header.seq`(`uint32`),模数 `2^31` → `2^32`**(回绕周期翻倍至约 13.6 年 @10Hz,且去掉有符号 Int32 约束)。
- **实现状态(v1.7)**:WSL 侧回绕安全比较/计数器 **M-A 已实现**;**固件侧 wire 行为(seq mod 2^32 回填)待 M-B 真机验证**(见任务卡 09/10)。注:此处 wire seq 回绕(2^32)与固件 MCU stamp 的 DWT CYCCNT 32 位回绕(~59.65s,§7.1 / 11 之 H3)是**两个独立的回绕问题**,勿混。
- **回绕策略(已随 exo_msgs 落地于 `link_health.py`)**:
  - **seq 语义 = 无符号回绕(wrap-around),按 `2^32` 取模**:到达 `2^32 - 1` 后下一个值回到 `0`。发送侧:`next = (cur + 1) mod 2^32`。`SEQ_MODULUS = 2^32`。
  - **所有「序列先后/匹配」判断必须用回绕安全比较**,不得用裸 `>` / `<`:
    - 用「序列号距离」语义(类 TCP 序号):`forward_distance(a, b) = (b - a) mod 2^32`;判断 b 是否在 a 之后的合理窗口内,而非直接比大小。
    - in-flight 配对用**精确相等**(值 N 对值 N),回绕不影响相等匹配;受回绕影响的只是「谁更新/谁迟到」这类有序判断。
  - **健康计数器**(`sent_count` / `lost_count` / `duplicate_count` / `matched_count` / `stale_duplicate_count`):监控侧用 Python int(无界 / ≥64 位)累加,服役年限内不溢出。这些计数器**独立于 wire seq 的 2^32 回绕**(它们不上 wire)。
  - **A7 回绕测试边界随之重标定到 `2^32`**:起始值置 `2^32` 边界附近、跨回绕点,验证取模回绕 / matched·lost·duplicate 判定 / 计数器不溢出仍正确。

### 7.7 诊断 topic `/exo/link_health`（v1.7 已实现，exo_msgs M-A）
- **已落地**(v1.7,原「规划」):`exo_cmd` 节点新增 `/exo/link_health` publisher(类型 `exo_msgs/msg/LinkHealth`,周期 ~1 Hz,参数 `link_health_period_s` 默认 1.0),把 tracker 的计数器 + RTT 滚动统计 + 对账标志打包发布。`header`(`std_msgs/Header`)的 `stamp` 用 **wall-clock**(诊断/bag 时间轴,**不参与 §7.1 RTT**)。
- **实现状态(v1.7)**:此 topic 为 **WSL 侧 `exo_cmd` 节点 M-A 已实现**;**固件侧不发布 LinkHealth**(诊断 topic 是 WSL 侧的),固件构建是否整包喂入 LinkHealth type support 属 M-B ROM 权衡(见 11 之 M4),不涉本节 wire 行为。
- **字段定稿**(`LinkHealth.msg`):
  ```
  std_msgs/Header header   # wall-clock stamp（诊断时间轴，不参与 RTT）
  uint64 sent
  uint64 matched
  uint64 lost
  uint64 duplicate
  uint64 stale_duplicate
  uint32 inflight
  float64 rtt_last_ms      # 最近一条 matched 的 RTT
  float64 rtt_p95_ms       # 有界滚动窗口内 p95（nearest-rank）
  float64 rtt_max_ms       # 窗口内最大
  bool reconciles          # sent==matched+lost+inflight，一眼看是否在悄悄掉东西
  ```
- **tracker 新增有界 RTT 滚动窗口**(`deque(maxlen=N)`,N=`rtt_window` 可配默认 1024):每条 matched echo 推入 `rtt_ms`,超窗淘汰最旧;维护并暴露 `rtt_last_ms` / `rtt_p95_ms`(nearest-rank `ceil(0.95·n)`) / `rtt_max_ms`,空窗口给确定占位 `0.0`。经 `rtt_stats()` 暴露,与既有 `counters()` / `reconciles()` 一同喂给 publisher。
- Phase A 用日志即可满足 7.1–7.5;此 topic 是为上位机/安全监控/数据记录预留的结构化出口。

### 7.8 验收点（给 Gill，全部 Phase A 可测，loopback 注入异常）
> 验证手段:loopback 节点(或测试夹具)可被指令注入「人为延迟 D ms」「按概率/指定 seq 丢弃 echo」「重复发送 echo」「不回 echo」,监控必须如实反应。优先用无 rclpy 依赖的逻辑单测(延续 G2 的 `test_roundtrip_logic.py` 风格),再辅以集成跑。

- **A1 RTT 可测(M1)**:正常 loopback 下,每条 matched echo 都能算出并记录 `rtt_ms`;注入固定延迟 `D` 后,记录的 RTT 应≈ `D`(在合理误差内)。无时间戳来源、用 wall clock 等做法判不通过。
- **A2 超限告警(M2)**:把注入延迟设为 `> rtt_warn_ms`(如 60 ms > 50 ms 占位值),监控必须产生 WARN,且含 seq 与实测 rtt;延迟设回阈值内则无告警(无误报)。阈值确为可配置(改 param 生效)。
- **A3 丢包可检(M3)**:让 loopback 对指定 seq **不回 echo**;经过 `rtt_deadline_ms` 后,该 seq 必须被判 lost、`lost_count` 精确 +1、并打告警。丢 k 条 ⇒ `lost_count` 增 k,不多不少。
- **A4 禁止静默驱逐(M4 / P1-1)**:制造 echo 全面迟到/积压(in-flight 堆积超出任何容量上界),验证**没有任何条目被无声删除**——被移除的条目要么 matched 要么计入 lost;`sent_count == matched_count + lost_count + inflight_size` 恒成立(对账等式)。代码层确认不存在 `min()` 式静默驱逐路径。
- **A5 重复 echo(M5 / P1-2)**:对同一 seq 重复回 echo,**不得**产生 UNMATCHED/错值告警;`duplicate_count` 精确累加;`matched_count` 不因重复而重复 +1。
  - **A5b 超窗重传(v1.6 / Gill 发现 #1)**:对一个**曾发出、但已挤出 `settled_window`** 的 seq 重发 echo ⇒ 判 `stale_duplicate`(不是 UNMATCHED),`stale_duplicate_count` 精确 +1(独立于 `duplicate_count`),至少 INFO 级可观测;对账恒等不变。
- **A6 错值仍报**:回一个从未发出的值 ⇒ 必须报 UNMATCHED/异常(确认 A5 没把真正的错值也放过)。
  - **A6b 越界 echo 仍报(v1.6 / Gill High;v1.7 域升级)**:回一个落在 `[0, 2^32)` 之外的值(负数 / ≥2^32)⇒ 必须报 UNMATCHED,**不得**被 wrap 取模 alias 成 `stale_duplicate` 漏报。
- **A7 回绕安全(P1-3;v1.7 边界=2^32)**:用单测把序列号置于 `2^32` 边界附近(注入接近上限的起始值并跨越回绕点),验证:① 发送计数按 `2^32` 取模回绕、不溢出;② matched/lost/duplicate 判定在回绕点仍正确(用距离比较,不被裸 `>` 误判);③ 健康计数器累加不发生溢出。
- **A9 seq/payload 解耦(v1.7)**:构造 `ExoStatus`,`header.seq` 固定、`payload` 取任意值(含与 seq 不等、负值、极值),验证配对只看 `header.seq`、payload 不影响 matched/duplicate/unmatched 判定。
- **A10 CRC 双路径(v1.7 / §7.9)**:`crc_enabled=True` 下喂一条 `header.crc` 与内容不符的 `ExoStatus` ⇒ `crc_mismatch_count` +1 + WARN,但 `header.seq` 仍喂 tracker(不阻断);`crc_enabled=False` 下坏 crc 不被判定(计数不动)。
- **A11 RTT 滚动窗口(v1.7 / §7.7)**:喂一串已知 RTT 的 matched 事件,验证 `rtt_last_ms`/`rtt_p95_ms`/`rtt_max_ms` 计算正确、窗口有界(超窗淘汰)、空窗口给确定占位。
- **A12 LinkHealth 发布(v1.7 / §7.7)**:验证 publisher 周期发 `LinkHealth`,字段与 tracker 快照一致(sent/matched/lost/duplicate/stale_duplicate/inflight/rtt_*/reconciles)。
- **A8 对账可观测**:任意时刻可读出 `sent / matched / lost / duplicate / stale_duplicate / inflight` 各量,且满足 A4 的对账等式(`stale_duplicate` 同 `duplicate`,不入对账恒等式);Gill 可据此一眼判断链路是否「悄悄掉东西」,并由 `stale_duplicate` 速率判断 `settled_window` 是否配小/链路是否积压。

### 7.9 应用级 CRC（v1.7 新增，自检开关，默认关）
- **定位(务必读清)**:`header.crc` 是**应用级打包自检开关**,**不是为防链路误码**。链路误码已由**帧层 CRC-16 + RELIABLE QoS**覆盖;本字段是为了抓**应用打包/序列化 bug**(如结构体字段写错偏移、两端字节序不一致、payload 被错位覆盖)的自检手段。把它当链路完整性保证会误导。
- **默认 disabled / 开发期可开**(参数 `crc_enabled`,默认 `False`):
  - 关:发送侧 `header.crc` 置 `0`,接收侧**不校验、不判定**。
  - 开:发送侧按下述规范填 `header.crc`;接收侧**先校验**,mismatch ⇒ `crc_mismatch_count` +1 + **WARN 告警,但不阻断**(仍把 `header.seq` 喂 tracker)。「自检失败」是可观测信号,不是丢帧理由。
- **算法 = CRC-32**(与 `crc` 字段 `uint32` 宽度一致;WSL 侧用 `zlib.crc32`)。
- **覆盖范围(启用时,契约写死)**:对 **`crc` 字段置 0 后**的 `seq‖stamp_mono_ns‖payload` **规范小端序字节流**计算(避免 CRC 自指):
  - `seq` → `uint32`(4 字节, LE)
  - `stamp_mono_ns` → `uint64`(8 字节, LE)
  - `payload` → `int32`(4 字节, LE, 补码)
  - 即 Python `struct.pack('<IQi', seq, stamp_mono_ns, payload)` 再 `zlib.crc32`。
- **范围边界**:M-A 只交付 **WSL 侧 Python 规范实现**。**两端字节序一致(F103 手算 CRC 与本规范逐字节对齐)是 M-B 联调事**,不在 M-A。
- **盲区声明(覆盖范围限定)**:CRC **仅自检 `(seq, stamp_mono_ns, payload)` 三元组的数值打包**,**不覆盖消息类型/方向**(不含 type name / topic / ExoCmd-vs-ExoStatus 区分)。即:CRC 能抓字段偏移错、字节序不一致、payload 错位覆盖等打包 bug;但**抓不出「类型张冠李戴/方向接反但三元组数值恰好一致」**这类错误——那类问题由 §1.1/§1.2 的 type support 匹配 + tracker 对账兜底,不靠 CRC。固件回填 ExoStatus 时,CRC 按**回填后的**三元组(seq 原样、stamp MCU 重盖、payload 原样)重算(见 11 之 H1)。
